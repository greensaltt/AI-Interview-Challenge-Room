# AI 面试闯关作战室 详细实施计划

## 1. 文档目标

本计划基于同目录下的 `design-document.md` 与 `tech-stack.md` 制定，面向 AI 开发者执行。目标是将项目拆解为一组小而具体、可独立验证的实施任务，确保：

- 每一步都有清晰输入、输出与边界
- 每一步都能被测试验证
- 先完成 MVP 主链路，再补充增强能力
- 不提前引入设计文档之外的复杂技术

## 2. 实施原则

- 只做 MVP 范围内功能
- 优先保证后端主链路闭环
- 所有数据库变更统一走迁移脚本
- Agent 只做工作流型实现，不做自治型实现
- 所有 AI 输出都必须有结构化校验与失败兜底
- 每完成一步必须先通过对应测试，再进入下一步

## 2.1 已确认执行约束

- 技术栈最终以 `tech-stack.md` 为准
- 模型服务默认使用 `DeepSeek`
- 对话主模型固定为 `DeepSeek-V3`
- 向量模型固定为 `deepseek-embedding-v2`
- 长耗时 AI 任务默认采用异步任务模式
- 异步任务交互固定为“提交任务 -> 返回 taskId -> 前端轮询状态”
- 注册方式固定为“用户名/邮箱 + 密码”
- 初始管理员账号通过数据库迁移脚本插入
- 初始管理员默认用户名为 `admin`
- 初始管理员默认邮箱为 `123@qq.com`
- 初始管理员默认密码为 `123456`
- 简历文件第一版存储在本地磁盘，并按分层目录管理
- 本地文件存储根路径位于项目根目录下
- 简历文件按 `userId` 分层存储，不再继续下分
- 简历文件仅支持 `PDF` 和 `DOCX`
- 单文件最大大小为 `10MB`
- 不允许上传同名简历文件
- 简历、JD、匹配分析等结果每次生成都创建新版本，默认读取最新有效版本
- 一个用户可以有多个学习计划，但同一岗位同一时刻仅允许一个有效计划
- 重新生成学习计划后，旧计划转为历史
- MVP 阶段题库仅供后台管理和 Agent 使用，不提供用户侧题库页面
- 模拟面试固定为“一次只问 1 题，最多 5-8 轮，用户可主动结束”
- 复盘评分维度固定为：技术准确性、表达清晰度、项目深度、问题分析能力、岗位匹配度
- Prompt 模板管理仅支持模板元数据管理和启用版本切换
- 历史报告对比分析放入第二阶段，不纳入当前 MVP
- 部署平台优先选择阿里云，第一版允许仅通过 IP 访问，不强制要求域名和 HTTPS

## 3. 交付顺序总览

建议按以下阶段推进：

1. 项目基础设施
2. 认证与权限
3. 核心业务数据模型
4. 简历与岗位链路
5. Agent 基础工作流
6. 学习计划与任务
7. 模拟面试与复盘
8. 后台管理
9. 部署与上线验证

## 4. 分步实施指令

### 步骤 01：建立工程骨架与目录结构

目标：

- 初始化 backend 与 frontend 工程
- 建立与技术栈文档一致的基础目录结构

给 AI 开发者的指令：

- 在 backend 中初始化 Spring Boot 单体工程
- 在 frontend 中初始化 Vue 3 工程
- 建立后端分层目录与前端页面目录
- 添加基础说明文档，说明如何启动本地开发环境

完成标准：

- backend 可以启动为空服务
- frontend 可以启动为空页面
- 仓库目录结构与设计文档保持一致

验证测试：

- 验证后端启动后可正常响应基础健康检查
- 验证前端开发服务可正常打开默认首页
- 验证项目根目录存在清晰的启动说明

### 步骤 02：建立本地基础设施运行环境

目标：

- 让 PostgreSQL、pgvector、Redis、后端、前端具备统一的本地运行方式

给 AI 开发者的指令：

- 配置本地开发所需的数据库与缓存运行方案
- 预留 pgvector 扩展初始化能力
- 统一环境变量命名规范
- 明确本地、测试、生产三类环境配置边界
- 为异步任务状态查询预留统一接口规范

完成标准：

- 本地开发环境可一键启动依赖服务
- 后端可连接 PostgreSQL 与 Redis
- 数据库支持向量扩展
- 前后端可支持基于 `taskId` 的轮询查询模式

验证测试：

- 验证 PostgreSQL 可连接并成功启用 pgvector
- 验证 Redis 可连接并可读写测试数据
- 验证后端启动时能正确读取本地环境配置

### 步骤 03：建立数据库迁移基线

目标：

- 确保后续所有数据结构变更都可追踪、可回放

给 AI 开发者的指令：

- 接入数据库迁移工具
- 建立首个迁移版本
- 先只创建最小系统基线表与扩展启用逻辑
- 在迁移脚本中插入一个初始管理员账号及对应角色关系
- 明确迁移文件命名规范和版本策略

完成标准：

- 新数据库可通过迁移工具初始化
- 迁移历史可在数据库中查询

验证测试：

- 验证在空数据库上执行迁移后无报错
- 验证重复执行迁移不会重复建表
- 验证迁移历史表中存在已执行记录
- 验证初始管理员账号和角色关系已正确写入
- 验证默认管理员用户名、邮箱和密码配置与约定一致

### 步骤 04：建立后端公共能力

目标：

- 为后续模块提供统一的接口、异常、日志与校验基础

给 AI 开发者的指令：

- 统一 API 返回结构
- 建立全局异常处理
- 接入参数校验
- 配置基础日志格式与请求追踪信息

完成标准：

- 所有基础接口返回风格一致
- 参数错误与系统错误可被清晰区分
- 日志中能追踪到请求级信息

验证测试：

- 验证正常请求返回统一成功结构
- 验证非法参数请求返回统一错误结构
- 验证日志中能看到请求路径、时间、结果状态

### 步骤 05：建立认证与权限数据模型

目标：

- 落地普通用户与管理员双角色模型

给 AI 开发者的指令：

- 建立用户、角色、用户角色关联等核心表
- 预留后续导师角色的扩展位
- 定义用户状态、角色状态和审计字段

完成标准：

- 用户与角色模型落库
- 可表达普通用户与管理员关系
- 数据结构支持后续 RBAC 扩展

验证测试：

- 验证迁移后存在用户与角色相关表
- 验证可插入普通用户和管理员测试数据
- 验证同一用户可关联多个角色

### 步骤 06：实现认证基础能力

目标：

- 完成注册、登录、退出、获取当前用户信息的基础链路

给 AI 开发者的指令：

- 实现注册、登录、退出、个人信息查询接口
- 注册标识支持用户名或邮箱
- 使用 JWT 作为基础认证方案
- 建立密码加密存储规则
- 明确未登录与无权限的错误返回

完成标准：

- 普通用户可完成完整登录链路
- 管理员与普通用户身份可区分

验证测试：

- 验证新用户可成功注册
- 验证可使用用户名登录
- 验证可使用邮箱登录
- 验证正确账号密码可登录并拿到令牌
- 验证错误密码登录失败
- 验证携带令牌可获取当前登录用户信息
- 验证未携带令牌访问受保护接口被拒绝

### 步骤 07：实现角色权限控制

目标：

- 让前后端都能识别普通用户与管理员可访问范围

给 AI 开发者的指令：

- 为后台接口加上管理员访问限制
- 为普通业务接口加上登录用户限制
- 明确权限失败时的标准返回

完成标准：

- 管理员接口不可被普通用户访问
- 普通用户接口不可被匿名访问

验证测试：

- 验证管理员可访问后台接口
- 验证普通用户访问后台接口被拒绝
- 验证匿名用户访问业务接口被拒绝

### 步骤 08：建立前端登录与路由守卫

目标：

- 让前端具备最小可用的登录态管理能力

给 AI 开发者的指令：

- 实现登录页、注册页、基础首页
- 建立前端登录态存储
- 配置路由守卫与角色判断
- 为后续业务页面预留基础布局

完成标准：

- 用户可在前端完成注册和登录
- 未登录用户无法进入受保护页面

验证测试：

- 验证登录成功后页面跳转到工作台
- 验证刷新后登录态仍能恢复
- 验证退出登录后访问受保护页面会被拦截

### 步骤 09：建立操作日志与审计基础

目标：

- 记录关键行为，支撑后台审计与问题排查

给 AI 开发者的指令：

- 建立操作日志表
- 定义关键操作记录范围，至少覆盖登录、创建、修改、删除、Agent 生成
- 在不影响主链路的前提下完成日志记录

完成标准：

- 关键操作可被记录与查询

验证测试：

- 验证登录行为会产生操作日志
- 验证创建与删除类接口会产生操作日志
- 验证日志中包含操作者、操作类型、时间、结果

### 步骤 10：建立简历管理数据模型

目标：

- 为简历上传、版本管理、结构化解析打基础

给 AI 开发者的指令：

- 建立简历表与简历解析结果表
- 区分原始文件信息、原始文本、结构化结果与当前启用状态
- 预留多版本简历能力
- 明确每次重新提取或解析都创建新版本，默认读取最新有效版本

完成标准：

- 同一用户可拥有多份简历
- 同一份简历有独立的解析结果记录

验证测试：

- 验证可为同一用户创建多份简历记录
- 验证每份简历可关联一条或多条解析结果
- 验证可标记某份简历为当前使用版本

### 步骤 11：实现简历上传基础链路

目标：

- 支持用户上传 PDF 或 DOCX 简历并持久化元数据

给 AI 开发者的指令：

- 实现简历上传接口
- 本地磁盘存储按分层目录组织文件
- 本地文件根路径位于项目根目录下
- 文件按 `userId` 分层存储
- 限制文件类型与大小
- 保存文件元数据与归属用户
- 明确本地开发与未来对象存储的抽象边界
- 拒绝同一用户上传同名简历文件

完成标准：

- 用户可上传简历文件
- 非法文件类型会被拒绝
- 超过 10MB 的文件会被拒绝
- 同名文件会被拒绝

验证测试：

- 验证上传合法 PDF 成功
- 验证上传合法 DOCX 成功
- 验证上传超限文件失败
- 验证上传非法类型文件失败
- 验证上传同名文件失败
- 验证用户只能看到自己的简历记录

### 步骤 12：实现简历列表与详情查询

目标：

- 让用户能管理自己的简历资产

给 AI 开发者的指令：

- 实现简历列表、详情、删除、切换当前简历接口
- 前端完成简历管理页面基础展示

完成标准：

- 用户能完成简历的基本管理

验证测试：

- 验证列表仅返回当前用户的简历
- 验证删除一份简历后列表正确更新
- 验证切换当前简历后状态持久化成功

目标：

- 为目标岗位管理与 JD 解析提供稳定的数据结构

给 AI 开发者的指令：

- 建立目标岗位表与 JD 解析结果表
- 保存岗位方向、公司、城市、级别、JD 原文等字段
- 区分原始输入与解析输出

完成标准：

- 一个用户可创建多个目标岗位
- 每个目标岗位可绑定 JD 文本与解析结果

验证测试：

- 验证可创建多个目标岗位
- 验证每个岗位可保存一份 JD 原文
- 验证解析结果可独立持久化

### 步骤 14：实现岗位与 JD 管理接口

目标：

- 完成目标岗位创建、查询、更新、删除的基础业务

给 AI 开发者的指令：

- 实现岗位与 JD 的增删改查接口
- 前端完成岗位目标页的基础表单与列表

完成标准：

- 用户可独立维护自己的岗位目标

验证测试：

- 验证创建岗位成功
- 验证更新岗位信息后查询结果正确
- 验证删除岗位后不可再次查询
- 验证用户不能访问其他用户的岗位数据

目标：

- 为专项练习与模拟面试提供结构化题目来源

给 AI 开发者的指令：

- 建立题库表
- 支持岗位方向、技能标签、难度、题目内容、参考答案等信息
- 为后续向量化与检索预留字段
- 题库当前仅服务于后台维护和 Agent 检索，不实现用户侧题库浏览能力

完成标准：

- 题库数据结构可支撑后台维护和检索

验证测试：

- 验证可插入不同岗位方向和难度的题目
- 验证标签字段可用于后续过滤查询

### 步骤 16：实现后台题库管理

目标：

- 让管理员可维护题库内容

给 AI 开发者的指令：

- 实现题库增删改查接口
- 实现后台题库管理页面
- 增加基础筛选能力

完成标准：

- 管理员可维护题库
- 普通用户不可访问后台题库管理能力

验证测试：

- 验证管理员可新增题目
- 验证管理员可按岗位方向或难度筛选题目
- 验证普通用户访问后台题库接口被拒绝

### 步骤 17：建立 Agent 基础接入层

目标：

- 形成统一的模型调用入口，避免 AI 调用散落在业务模块中

给 AI 开发者的指令：

- 接入 Spring AI
- 建立统一的模型配置、调用封装、错误处理和超时策略
- 抽象聊天模型与向量模型配置
- 建立 Prompt 模板管理目录
- 默认接入 DeepSeek 模型服务
- 对话主模型固定使用 `DeepSeek-V3`
- 向量模型固定使用 `deepseek-embedding-v2`

完成标准：

- 后端具备统一的 AI 调用能力
- 后续工作流可复用同一套调用封装

验证测试：

- 验证模型连接配置生效
- 验证可执行一次最小 AI 调用并拿到结果
- 验证模型调用失败时返回可识别错误而不是系统崩溃

### 步骤 18：实现简历文本提取链路

目标：

- 将上传的简历文件转为可用于 Agent 分析的文本内容

给 AI 开发者的指令：

- 为 PDF 与 DOCX 建立文本提取流程
- 提取结果保存到简历相关记录中
- 若提取失败，保留失败状态与原因

完成标准：

- 合法简历文件可被转换为文本

验证测试：

- 验证 PDF 简历可提取文本
- 验证 DOCX 简历可提取文本
- 验证提取失败时记录错误状态

### 步骤 19：实现简历结构化解析工作流

目标：

- 将简历文本转为结构化信息，供后续 JD 匹配与面试使用

给 AI 开发者的指令：

- 设计简历结构化输出格式
- 实现结构化解析工作流
- 将原始文本、结构化结果、版本信息持久化
- 加入结果校验与失败兜底
- 每次重新解析都创建新版本记录，默认读取最新有效版本

- 验证结构化结果可通过字段校验
- 验证模型返回异常格式时系统能给出失败状态与兜底信息

### 步骤 20：实现 JD 解析工作流

目标：

- 将 JD 原文解析为结构化岗位需求

给 AI 开发者的指令：

- 设计 JD 结构化输出格式
- 实现 JD 解析工作流
- 保存岗位职责、技能要求、加分项、标签等结果
- 加入结构校验与失败兜底

- 每个岗位 JD 都可生成结构化解析结果

验证测试：

- 验证解析结果包含岗位名称、职责、技能要求、标签
- 验证空白或无效 JD 输入会被拒绝或正确标记失败
- 验证重复解析会生成新版本或正确覆盖策略

### 步骤 21：实现简历与 JD 匹配分析工作流

目标：

- 产出用户最关心的优势、短板与补强方向

给 AI 开发者的指令：

- 设计匹配分析输出格式
- 根据简历结构化结果与 JD 结构化结果生成匹配分析
- 保存匹配度、优势项、缺口项、建议补强方向

验证测试：

- 验证当简历或 JD 缺失时不允许发起分析
- 验证分析结果可在后续学习计划生成中被正确引用

### 步骤 22：建立学习计划与任务数据模型

- 支持计划周期、计划状态、任务状态、任务备注

- 约束同一岗位同一时刻仅有一个有效计划

验证测试：

- 验证可创建计划并关联多条任务
- 验证任务状态可表达待完成、进行中、已完成、已跳过
- 验证同一岗位不能同时存在多个有效计划

### 步骤 23：实现学习计划生成工作流

- 基于岗位差距分析生成阶段性训练计划

- 将匹配分析结果转化为学习目标和训练重点
- 输出结构化计划并落库
- 当同一岗位重新生成计划时，将旧计划转为历史

完成标准：

- 用户可针对某个岗位生成一份学习计划

验证测试：

- 验证三种计划周期都可生成有效计划
- 验证计划中包含学习目标、重点方向、每日任务草案
- 验证缺少匹配分析结果时不能直接生成计划
- 验证重新生成计划后旧计划状态已转为历史

### 步骤 24：实现每日任务生成与状态流转

## 5. 推荐并行策略

为了提高效率，建议按以下方式分工：

- AI 开发者 A：后端基础设施、认证权限、数据库迁移
- AI 开发者 B：简历、岗位、计划、任务等核心业务模块
- AI 开发者 C：Agent 工作流、向量检索、复盘链路
- AI 开发者 D：前端页面、联调、后台管理页面

并行注意事项：

- 所有人统一以 `memory-bank/design-document.md` 和 `memory-bank/tech-stack.md` 为准
- 数据表变更必须先同步迁移方案
- 统一接口返回格式与错误码
- Agent 输出结构需先约定再落实现

## 7. 明确禁止事项

- 不要提前引入微服务拆分
- 不要接入支付系统
- 不要做联网搜索
- 不要做视频面试

- 不要把 Agent 实现为不可控的自治模式
- 不要在缺乏结构校验时直接落库存储 AI 输出
