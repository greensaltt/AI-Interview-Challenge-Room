## 2. 项目概述

### 2.1 项目定位

AI 面试闯关作战室是一个面向实习/校招大学生的 Web 求职训练平台。产品通过 Agent 能力将传统的求职准备流程进行结构化串联，帮助用户完成从岗位理解、简历诊断、学习规划到模拟面试和复盘成长的完整闭环。

本项目的目标不是做一个泛聊天机器人，而是做一个具备明确业务流程、可展示 Java 工程能力与大模型应用能力的作品集级 Agent 项目。

### 2.2 目标用户

- 准备实习的大学生
- 准备校招的大学生

### 2.3 产品目标

- 帮助用户围绕目标岗位完成系统化面试训练
- 通过任务化、闯关化设计提升训练体验和持续使用意愿
- 展示完整后端业务系统能力
- 展示基于 Spring AI 的工作流型 Agent 落地能力
- 支持低成本上线部署，适合作为个人作品集长期展示

### 2.4 非目标

以下内容不作为 MVP 目标：

- 真正接入支付、退款、会员结算
- 联网搜索与实时外部数据抓取
- 多角色复杂协作体系，如导师端、企业端
- 视频面试
- 高并发商业化能力建设
- 完全自治型 Agent

## 4. 用户角色与权限

### 4.1 普通用户

普通用户可以：

- 注册、登录、退出登录、修改密码、注销账号
- 上传和管理简历
- 新建目标岗位和导入 JD
- 查看 JD 解析与简历诊断结果
- 生成学习计划与每日任务
- 发起模拟面试
- 查看历史面试记录和复盘报告
- 查看题库和推荐练习内容

### 4.2 管理员

管理员可以：

- 管理用户信息
- 管理岗位模板和题库内容
- 查看面试记录与生成报告
- 管理知识库基础内容
- 查看系统日志和关键操作记录
- 配置基础 Prompt 模板或策略参数

### 4.3 后续保留角色

- 导师：作为未来扩展角色保留，后续可接入人工点评、计划干预、面试复查等能力

## 5. 核心业务流程

### 5.1 主流程

1. 用户注册并登录系统
2. 用户上传简历
3. 用户创建目标岗位并填写或导入 JD
4. Agent 对 JD 和简历进行解析与能力差距分析
5. 系统生成阶段性学习计划和每日任务
6. 用户开始模拟面试
7. Agent 根据岗位、JD、简历、历史记录发起文本面试
8. 面试结束后系统生成复盘报告
9. 用户根据报告继续完成后续任务，形成成长闭环

### 5.2 面试训练闭环

1. 岗位目标设定
2. 简历与 JD 匹配分析
3. 学习计划生成
4. 每日训练任务执行
5. 模拟面试
6. 面后复盘
7. 再训练与再面试

## 6. 功能设计

### 6.1 用户中心

功能点：

- 账号注册、登录、退出
- JWT 鉴权
- 个人信息维护
- 修改密码
- 注销账号
- 登录日志与操作日志记录

设计要点：

- 支持普通用户和管理员角色
- 为后续导师角色预留 RBAC 扩展位

### 6.2 简历管理

功能点：

- 上传简历文件
- 简历文本提取与结构化解析
- 简历版本管理
- 选择某份简历作为当前求职简历

设计要点：

- 支持 PDF / DOCX
- 解析结果需形成结构化字段，便于后续 Agent 使用
- 原始文本与结构化结果均需保存

### 6.3 岗位 / JD 管理

功能点：

- 新建目标岗位
- 手动粘贴 JD
- 导入外部 JD 文本
- 保存岗位方向、城市、公司、级别等信息
- 对 JD 进行解析，提取技能要求、职责、加分项

设计要点：

- JD 解析结果需转化为结构化标签
- 结构化结果可直接作为学习计划和面试题生成的输入

### 6.4 学习计划

功能点：

- 基于简历与 JD 差距生成阶段性学习计划
- 生成计划周期，如 7 天 / 14 天 / 21 天
- 支持重新生成或局部调整
- 记录计划执行状态

设计要点：

- 计划应包含主题、目标、建议资料方向、练习方式
- 计划结构化保存，便于任务拆分和进度统计

### 6.5 每日任务

功能点：

- 根据学习计划自动拆解每日任务
- 任务状态流转：待完成、进行中、已完成、已跳过
- 支持补充个人备注
- 支持基于任务内容进入专项练习

设计要点：

- 任务必须可追踪、可量化
- 任务应尽量短小，便于形成日常使用习惯

### 6.6 题库

功能点：

- 管理员维护题库
- 按岗位方向、技能标签、难度分类
- 为模拟面试和专项训练提供题目来源

设计要点：

- 题库既是后台内容模块，也是 Agent 的知识来源之一
- 初期题库可由管理员手动录入，不依赖联网抓取

### 6.7 模拟面试

功能点：

- 用户选择目标岗位发起面试
- Agent 结合 JD、简历、计划、历史记录生成问题
- 支持多轮追问
- 支持按轮次保存问答记录
- 用户可主动结束面试

设计要点：

- 第一版为纯文本面试
- 后续预留语音输入、语音播报、视频面试扩展位
- 面试流程采用工作流型 Agent，不做完全自治

### 6.8 复盘报告

功能点：

- 面试结束后生成结构化复盘报告
- 输出总体表现、能力维度评价、薄弱点、改进建议
- 支持引用面试记录中的具体片段做针对性分析
- 支持与历史报告做对比

设计要点：

- 报告不只做泛泛总结，需要能够引用关键问答片段
- 支持数据化评分 + Agent 文字分析相结合

### 6.9 后台管理

功能点：

- 用户管理
- 岗位模板管理
- 题库管理
- 面试记录查询
- 报告查询
- Prompt 模板基础管理
- 操作日志查看

设计要点：

- 后台既是内容配置中心，也是系统运营入口
- 应尽量覆盖作品集中常见的管理后台能力

## 7. Agent 设计

### 7.1 Agent 形态

本项目采用工作流型 Agent，而非自治型 Agent。

原因如下：

- 流程边界更清晰，便于控制输出质量
- 更适合作品集项目展示业务建模与工程能力
- 更利于调试、日志追踪和问题定位
- 更符合 Spring AI 在 Java 应用中的落地方式

### 7.2 Agent 能力范围

MVP 阶段主要覆盖：

- JD 解析
- 简历结构化解析
- 简历与 JD 匹配分析
- 学习计划生成
- 每日任务生成
- 模拟面试问题生成
- 个性化追问
- 面后复盘总结
- 基于知识库的检索增强

### 7.3 Agent 工作流

#### 工作流一：JD 解析

输入：

- JD 原始文本

输出：

- 岗位名称
- 岗位职责
- 核心技能要求
- 加分项
- 难度等级
- 技能标签

#### 工作流二：简历诊断

输入：

- 简历结构化信息
- JD 结构化结果

输出：

- 匹配度分析
- 优势项
- 缺口项
- 建议补强方向

#### 工作流三：学习计划生成

输入：

- 简历诊断结果
- 用户目标岗位
- 计划周期

输出：

- 周期化学习目标
- 每日任务清单
- 重点知识方向

#### 工作流四：模拟面试

输入：

- 用户简历
- 目标岗位
- JD 结构化信息
- 历史面试表现
- 题库与知识库结果

输出：

- 首轮问题
- 追问问题
- 单轮评价
- 整场面试记录

#### 工作流五：面后复盘

输入：

- 面试完整记录
- 目标岗位信息
- 历史面试记录

输出：

- 总体评价
- 能力分项评价
- 薄弱点总结
- 改进建议
- 引用的关键问答片段

### 7.4 AI 输出控制策略

为保证结果可落库、可展示、可追踪，AI 能力应优先采用以下工程化手段：

- 结构化输出
- Prompt 模板化
- 固定工作流节点
- 失败重试与兜底模板
- 关键结果持久化

## 8. 知识库与检索设计

### 8.1 MVP 知识源

第一版知识库仅包含以下内容：

- 平台内置题库
- 用户上传的简历
- 用户导入的 JD
- 用户历史面试记录
- 用户历史复盘报告

### 8.2 检索目标

检索的目的不是回答开放世界问题，而是增强以下业务能力：

- 面试出题更贴近目标岗位
- 追问更贴近用户历史表现
- 复盘更能关联上下文
- 学习计划更具针对性

### 8.3 向量化建议

- 采用 PostgreSQL + pgvector 作为低成本向量存储方案
- 对题库、JD、简历文本、历史问答片段进行切分和向量化
- 检索结果作为 Agent 上下文输入，而非直接展示给用户

## 9. 技术架构设计

### 9.1 技术选型

后端：

- Java 17+
- Spring Boot
- Spring AI
- Spring Security
- MyBatis-Plus 或 Spring Data JPA
- PostgreSQL
- pgvector
- Redis

前端：

- Vue 3
- Vite
- Element Plus
- Pinia
- Vue Router

部署：

- Docker Compose
- Linux 云服务器
- Nginx

### 9.2 总体架构

系统整体采用前后端分离架构：

1. 前端 Web 负责用户交互、数据展示、面试对话界面
2. 后端业务服务负责用户、岗位、题库、计划、任务、面试、报告等核心业务
3. Agent 服务能力通过 Spring AI 嵌入后端业务服务中
4. PostgreSQL 负责业务数据和向量数据存储
5. Redis 负责缓存、限流、短期会话状态等能力
6. 外部大模型服务通过统一接口接入

### 9.3 模块划分建议

后端模块建议：

- `auth`：认证鉴权
- `user`：用户中心
- `resume`：简历管理
- `job`：岗位与 JD 管理
- `question-bank`：题库管理
- `plan`：学习计划
- `task`：每日任务
- `interview`：模拟面试
- `report`：复盘报告
- `agent`：Prompt、工作流、模型调用、结构化输出
- `knowledge`：知识库与检索
- `admin`：后台管理
- `log`：操作日志与审计日志

前端页面建议：

- 登录注册页
- 用户工作台
- 简历管理页
- 岗位目标页
- 学习计划页
- 每日任务页
- 模拟面试页
- 复盘报告页
- 后台管理页

## 10. 数据模型设计

### 10.1 核心实体

建议优先设计以下核心表：

- `sys_user`：用户表
- `sys_role`：角色表
- `sys_user_role`：用户角色关联表
- `user_resume`：简历表
- `job_target`：目标岗位表
- `job_jd_parse_result`：JD 解析结果表
- `resume_parse_result`：简历解析结果表
- `resume_jd_match_result`：简历与 JD 匹配结果表
- `study_plan`：学习计划表
- `study_task`：每日任务表
- `question_bank`：题库表
- `mock_interview`：模拟面试表
- `mock_interview_round`：面试轮次问答表
- `interview_report`：复盘报告表
- `knowledge_chunk`：知识切片表
- `vector_index_ref`：向量索引引用表
- `operation_log`：操作日志表

### 10.2 关键关系

- 一个用户可拥有多份简历
- 一个用户可维护多个目标岗位
- 一个目标岗位对应一份 JD 原文和一份或多份解析结果
- 一个学习计划对应多个任务
- 一场模拟面试对应多轮问答
- 一场模拟面试对应一份复盘报告
- 题库可按岗位方向和技能标签归类

### 10.3 数据设计原则

- 原始文本与结构化结果分开存储
- AI 输出结果尽量保留版本信息
- 长文本记录与摘要字段并存，兼顾可读性和检索效率
- 为后续语音和视频能力预留媒体字段

## 11. 接口设计原则

### 11.1 API 风格

- 采用 RESTful API
- 统一返回结构
- 统一异常码与错误信息
- 支持分页、条件筛选、按时间倒序查询

### 11.2 重点接口分组

- 认证接口
- 用户中心接口
- 简历上传与查询接口
- 岗位创建与 JD 解析接口
- 学习计划生成与查询接口
- 每日任务流转接口
- 模拟面试接口
- 复盘报告生成与查询接口
- 后台管理接口

## 12. 非功能设计

### 12.1 可用性

- 优先保证主链路可跑通
- Agent 失败时提供兜底提示，不阻断核心业务
- 对长耗时任务采用异步化处理

### 12.2 可维护性

- 模块分层清晰
- Prompt 模板集中管理
- 关键 Agent 结果持久化，便于排查
- 日志链路完整

### 12.3 安全性

- JWT 鉴权
- RBAC 权限控制
- 关键接口限流
- 文件上传类型校验
- 敏感操作日志记录

### 12.4 成本控制

- 使用 PostgreSQL + pgvector，避免额外向量数据库成本
- 单机 Docker Compose 部署，降低云资源成本
- 初期不引入复杂微服务架构
- 语音能力延后实现，避免前期音视频成本过高

## 13. 部署方案

### 13.1 部署目标

- 可在线访问
- 成本尽量低
- 便于个人维护和演示

### 13.2 MVP 推荐部署方式

推荐采用单机部署：

- 1 台 Linux 云服务器
- Docker Compose 管理前端、后端、PostgreSQL、Redis、Nginx

推荐原因：

- 成本低
- 结构简单
- 适合作品集项目长期展示
- 便于后续扩展语音服务

### 13.3 部署建议

- 前端打包为静态资源，由 Nginx 提供服务
- 后端以 Spring Boot 容器运行
- PostgreSQL 同时承担业务库与向量库能力
- Redis 用于缓存与临时状态管理
- 域名和 HTTPS 可在上线阶段补齐

## 15. 风险与应对

### 15.1 范围膨胀风险

风险：

- 功能过多导致项目迟迟无法完成

应对：

- 严格按照 MVP 范围推进
- 把语音、视频、支付、导师体系放到后续阶段

### 15.2 AI 结果不稳定风险

风险：

- 输出格式不稳定，影响落库和前端展示

应对：

- 使用结构化输出
- 控制 Prompt 和工作流节点
- 增加结果校验和兜底逻辑

### 15.3 成本风险

风险：

- 模型调用和部署成本过高

应对：

- 优先做文本链路
- 统一模型接入层
- 使用单机部署和 pgvector 降低基础设施成本

### 15.4 前端复杂度风险

风险：

- 前端经验不足导致界面开发拖慢进度

应对：

- 采用 Vue 3 + Element Plus
- 先完成后台和主业务页面，避免过早追求复杂交互

## 16. 设计假设

以下内容当前作为默认设计假设，后续如有需要可再调整：

- 前端采用 Vue 3 + Element Plus
- 后端采用 Spring Boot + Spring AI
- 模拟面试第一版仅支持文本
- 语音面试作为下一阶段能力预留
- 知识库不接联网搜索
- 支付与会员仅保留未来扩展空间
- 管理员后台包含基础 Prompt 配置能力

## 17. 成功标准

当项目满足以下条件时，可视为达到作品集 MVP 目标：

- 用户可完成从简历上传到模拟面试复盘的完整闭环
- 管理员可维护题库、岗位模板和基础系统内容
- Agent 能稳定输出结构化诊断结果和复盘结果
- 系统可在线部署访问
- 项目能清晰展示 Java 后端基础能力与大模型应用能力
